# Application Architecture

## High-Level App Architecture

The LangGraph Autonomous Dev Team application is designed as a backend orchestration service that manages a stateful, multi-agent software development lifecycle.

The system is exposed via a **FastAPI** web service which provides webhooks to initiate the workflow and endpoints to poll for status or provide human input. Internally, the workflow is modeled as a **LangGraph StateGraph**.

### The Graph Workflow
1. **PO Agent:** Receives the original ticket, extracts Use Cases via LLM structured outputs.
2. **Dev Lead Agent:** Evaluates Use Cases and generates technical tasks (Frontend, Backend, DevOps).
3. **DevOps Agent:** Scaffolds the application structure based on the Dev Lead's tasks.
4. **Developer Agents:** Simulates code implementation and PR creation.
5. **QA Agent:** Reviews and merges the mock PRs.

The state of the graph is persistently managed using **SQLite** via `langgraph.checkpoint.sqlite.SqliteSaver`, which allows the workflow to be paused (e.g., waiting for human clarification) and safely resumed.

## Core Technologies Used

### Orchestration Backend (This Repository)
- **FastAPI:** For the REST API layer.
- **LangGraph (Python):** For defining and running the multi-agent graph.
- **SQLite:** For localized state check-pointing and persistence.
- **LangChain & OpenAI:** For LLM decision making.

### Scaffolded Frontend Projects (Managed by DevOps Agent)
In adherence to the global project rules, whenever the **DevOps Agent** scaffolds a *new frontend project*, it strictly enforces the following mandatory stack:
- **React** – Core UI library
- **Next.js** – Application framework (routing, SSR/SSG, app structure)
- **Supabase** – Backend-as-a-service (auth, database, storage)
- **Tailwind CSS** – Styling and design system

## Technologies Explicitly Avoided
- Alternative frontend frameworks (e.g., Vue, Svelte, Angular) are never scaffolded.
- Alternative CSS frameworks (e.g., Bootstrap, MUI) are avoided in favor of Tailwind CSS.
- The use of browser-native blocking dialogs (`alert()`, `confirm()`, `prompt()`) is strictly prohibited in scaffolded code.
- Alternative Backend-as-a-Service tools (e.g., Firebase) are avoided in favor of Supabase.

## Data Flow and Patterns

1. **Input:** A JSON payload representing a Linear ticket is sent to `/webhook/ticket`.
2. **State Initialization:** The ticket details are loaded into the `AppState` (a TypedDict), and a unique `thread_id` is generated.
3. **Agent Traversal:** The state is passed from node to node. The `ticket_store` (abstracted) maintains a parallel record of tasks (simulating Linear). The `version_control` (abstracted) maintains a record of code branches and PRs (simulating GitHub).
4. **Global Required Utility / Pattern:** Any frontend code generated by the Developer or DevOps agents is instructed to include, implement, or adapt the required utility pattern defined globally for the project (e.g., robust UI fallbacks and mock data rendering when Supabase is disconnected).
5. **Human-in-the-Loop:** If the graph encounters a `needs_human_input` flag, execution pauses. An external system can hit `/workflow/{thread_id}/human-input` with the required clarification to resume execution.

## Frontend and Backend (Supabase) Interaction
While this LangGraph API does not connect to Supabase directly, it acts as the orchestrator that *builds* systems that do. The DevOps agent scaffolds the Next.js frontend with Supabase SDKs, setting up fallback static mock data if the Supabase environment variables are missing, ensuring the scaffolded app is always locally runnable for UI/UX review.
